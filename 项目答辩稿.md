# 中国近五年天气抓取与可视化分析项目答辩稿

## 1. 开场介绍

各位老师好，我的项目名称是《中国近五年天气抓取与可视化分析》。

这个项目的核心目标，是围绕中国主要城市近五年的历史天气数据，完成从数据抓取、清洗处理、统计分析，到交互式可视化展示的一整套流程。

我希望这个项目不仅能完成基础的数据分析任务，还能尽量做成一个接近成品化的数据展示系统，而不是简单输出几张静态图。

## 2. 项目背景与研究意义

天气数据本身具有明显的时间序列特征、区域分布特征和多指标耦合特征，例如气温、降水、风速之间会随季节和地区出现不同规律。

如果只使用表格或者单张图表，很难完整表达这些变化关系。

因此，这个项目的意义主要有三点：

1. 把近五年的天气数据做成可视化分析结果，提升数据可理解性。
2. 通过时间轴地图和趋势联动，让空间变化和时间变化能够结合起来观察。
3. 完整演示一个数据项目从抓取到展示的工程实现过程。

## 3. 项目目标

本项目主要实现了以下几个目标：

1. 自动抓取中国主要城市近五年的历史天气数据。
2. 对数据进行清洗、补全、聚合，形成日度和月度数据表。
3. 构建一套交互式可视化大屏。
4. 支持地图时间轴、指标切换、省份联动趋势图等交互功能。
5. 输出文字分析摘要，形成完整的项目交付结果。

## 4. 技术路线

这个项目的技术路线可以分成四层。

第一层是数据抓取层。

项目当前使用的是 `Open-Meteo Historical Weather API`，按城市经纬度抓取近五年的历史天气数据。

第二层是数据处理层。

主要使用 `pandas` 和 `numpy` 完成：

- 字段清洗
- 日期拆分
- 缺失值插值
- 日度到月度聚合
- 趋势斜率计算
- 舒适度指标构造

第三层是可视化层。

常规图表使用 `pyecharts` 生成，地图和复杂联动部分使用原生 ECharts 脚本进行增强。

第四层是展示层。

最终输出为一个固定布局的 HTML 可视化大屏，以及一份 Markdown 文本报告。

## 5. 数据内容与采样范围
当前抓取链路最近又做了一轮稳定性增强。
现在天气接口和行政区划接口都统一补充了重试、退避以及可中断等待，所以在网络波动或瞬时限流时，整体失败率比之前更低。
另外，行政区划接口如果只是某一个省失败，程序现在只会回退这个省，不再因为单省失败整批退回旧样本。

项目目前覆盖中国主要区域的代表城市，包括：

- 华北
- 东北
- 华东
- 华中
- 华南
- 西南
- 西北

后续还补充了台湾的三个代表城市：

- 台北
- 台中
- 高雄

当前使用的天气字段主要有：

- 平均气温
- 最高气温
- 最低气温
- 体感温度
- 降水量
- 风速
- 阵风
- 风向
- 天气代码

现在代码已经进一步升级为：

- 先按省份加载行政区划数据
- 再自动提取地级市名称、行政区代码和中心点坐标
- 最终按地级市构建抓取列表

也就是说，项目已经具备向完整地级市数据扩展的能力。

## 6. 数据处理流程
在数据处理流程里，我还把失败重抓这条链路修正了。
当前失败城市会直接写入 `.api_cache/failed_cities.json`，而“仅重抓失败数据”按钮现在直接读取这份失败清单，不再重新根据最新动态城市列表做匹配。
这样可以避免页面点了失败重抓后，终端没有实际执行、页面却很快弹出失败提示的问题。

整个数据处理流程主要包括以下步骤：

1. 根据运行日期自动计算近五年的时间区间。
2. 按城市抓取历史天气数据。
3. 将抓取结果缓存到本地，避免重复请求。
4. 对字段进行统一命名和格式转换。
5. 对缺失值进行插值补全。
6. 生成年、月、年-月等分析字段。
7. 聚合形成月度数据。
8. 构造舒适度、温差和城市升温趋势等衍生指标。

最终输出两张核心数据表：

- 日度数据表
- 月度聚合表

## 7. 可视化系统设计
另外，服务启动逻辑也做了调整。
现在 `--serve-dashboard` 默认会优先快速启动本地控制服务，不再像之前那样先阻塞执行整套全量抓取，所以打开本地服务的等待时间明显缩短。
地图上方按钮布局也重新做了分组整理，交互区会更规整。

除地图之外，其余板块我也根据当前真实数据状态重新做了口径调整。
顶部摘要卡片当前展示的是覆盖城市数、覆盖省级区域数、最新月份和待重抓城市数，不再使用旧版的最热、最冷这类容易被误解成完整全国结果的口径。
其他趋势图、热力图、雷达图和散点图的说明，也改成了“当前样本”口径，更符合实际已经落地的数据范围。
当前大屏在地图区域又进一步补充了本地数据控制能力：
- 可以直接点击“更新全部数据”
- 可以直接点击“仅重抓失败数据”
- 可以在任务运行时点击“中断任务”
- 页面会同步显示总进度、成功数、失败数和当前处理城市

本项目的大屏不是默认拼图页，而是专门做了固定栅格布局。

主要包含以下模块：

### 7.1 顶部摘要卡片

展示近五年核心结论，例如：

- 覆盖城市数
- 覆盖省级区域数
- 最新月份
- 待重抓城市数

### 7.2 中国分省气候时序地图

这是项目的核心模块之一。

主要功能包括：

- 拖动时间轴查看不同月份
- 切换平均气温、降水量、平均风速
- 自动播放
- 点击省份后联动趋势图
- 双击省份后进入该省城市视图

当前自动播放逻辑已经改为直接切换月份，不再执行插值过渡。

### 7.3 省份联动趋势图

功能包括：

- 默认显示全国平均趋势
- 点击地图省份后切换为该省趋势
- 点击趋势图某个月份后反向切换地图时间

### 7.4 其他分析图表

包括：

- 气候空间分布图
- 当前样本月度温度趋势图
- 区域热力图
- 升温趋势排行
- 区域雷达图
- 温度与降水散点图

## 8. 项目中的关键工程问题与解决过程

这个项目不只是完成了功能，也经历了很多实际工程问题。

### 8.1 数据源切换

项目一开始尝试使用 `Meteostat`，但后来遇到了几个问题：

- Python 3.14 兼容性问题
- 缓存目录并发报错
- 数据源连接被拒绝

所以后续改成了 `Open-Meteo Historical Weather API`。

### 8.2 限流问题

公共接口在高并发下容易出现 `429 Too Many Requests`。

我做了以下处理：

- 降低默认线程数
- 增加重试和退避等待
- 推荐使用单线程运行

### 8.3 缓存失效导致结果残缺

后期用户环境里又出现了一个新问题：

- 运行日期变化后，缓存文件名变化
- 同时网络请求失败
- 结果被覆盖成只有部分城市的数据

我后来增加了“历史缓存回退机制”：

- 如果当天抓取失败，就自动回退到最近一次成功缓存

这个改动非常关键，直接提升了项目稳定性。

### 8.4 地图省名映射问题

地图底图使用的是标准省级行政区全名，但数据里很多是简称。

例如：

- 北京 对应 北京市
- 新疆 对应 新疆维吾尔自治区
- 广西 对应 广西壮族自治区

因此我增加了省名映射表，保证地图可以正确着色。

### 8.5 趋势图高亮功能的取舍

项目后期曾多次尝试给趋势图加入“当前月份高亮线”和“高亮点”，但由于用户本地浏览器环境和图表渲染链路不稳定，这个功能一直存在反复失效的问题。

我前后尝试了多种实现方式，包括：

- `markLine`
- 自定义图形层
- 渲染完成后再画
- 第二系列高亮点
- 第三系列竖线

最终结论是：

- 这个细节效果收益有限
- 但不稳定性较高

所以我最终选择移除这个高亮功能，保留主功能稳定性。

这是一个典型的工程取舍案例。

### 8.6 地级市扩展后的缓存冲突处理

当地级市数据开始引入后，不同省份可能出现重名城市。

如果缓存文件仍然只按城市名命名，就可能发生相互覆盖。因此当前缓存文件命名已经改为优先使用行政区代码，而不是只用城市名。

如果按版本迭代来概括，这一轮可以补充为：

`V15：地级市抓取与省内钻取升级`

### 8.7 大屏按钮控制与任务中断
后续我又继续修正了一轮细节：
- 中断按钮现在会即时显示“已发送中断请求”
- 等待与重试阶段也能响应中断
- 启动任务与点击中断之间的竞态问题也做了修正
为了避免每次都回到命令行重新执行整批抓取，我又补了一轮大屏控制能力。
现在页面里可以直接触发全量更新，也可以只重抓上一次失败的城市。
同时，为了避免用户连续点击导致重复启动多个抓取任务，我在任务运行时会临时禁用两个更新按钮，只保留“中断任务”按钮可用。
如果用户主动中断，系统会尽量停止未开始的城市任务，并把失败或未完成的城市保留到失败清单里，后续可以继续用“仅重抓失败数据”补抓，而不是从头全部重跑。
为了方便本地演示，我还补了 `start_dashboard.bat`，双击后会自动启动本地控制服务并打开大屏页面。

## 10. 当前不足
补充一点：
- 虽然非地图板块已经改成真实样本口径，但只要 `processed` 结果还没补齐，全国统计结论仍然会继续变化。

目前项目仍有几个不足：

1. 台湾数据已加入采样列表，但由于当前代理环境问题尚未成功抓取。
2. 地图底图仍依赖在线 GeoJSON，请求失败时地图无法正常显示。
3. 公共历史天气接口稳定性受网络环境影响较大。
4. 趋势图高亮功能最终被移除，没有保留到最终版本。
5. 当前仓库里的处理数据未必已经是完整地级市结果，这取决于最近一次全量抓取是否成功。
6. 如果网络或代理异常，虽然代码已经支持地级市抓取，但最终仍可能只能回退到旧样本数据。

补充说明：
- 大屏中的数据更新按钮依赖本地 Python 控制服务，服务未启动时页面地址和按钮都不可用

## 11. 后续优化方向

如果继续扩展，我认为最值得做的有四项：

1. 将地图底图本地化，做成离线可用版本。
2. 为台湾城市补充离线缓存或备用数据。
3. 增加顶部卡片的动态联动。
4. 增加极端天气事件分析和异常年份分析模块。
